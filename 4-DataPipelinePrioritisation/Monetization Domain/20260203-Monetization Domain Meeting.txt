Summary
Data Pipeline Migration Requirements
The team discussed the requirements for migrating three data pipelines related to monetization, including merchant fee reporting, payout offers, and coupon disputes. Kaihao explained that the new data model will consist of three separate databases: a Conformed layer, a Metrics layer, and a Semantic layer, with data flowing from the first to the third. The team agreed that while some current tables will be directly migrated to the Conformed layer, others will be remodeled to better fit the new data structure. The project aims to not only migrate existing data but also review and potentially redesign metrics with SMEs, making the data more intuitive for consumption by LMs.
DBT Migration Strategy Discussion
Mazen and Kaihao discussed the migration of data models and DBT (Data Build Tool) projects. They agreed that new models would be created rather than reusing old ones, and historical data would be migrated to new tables. Kaihao confirmed that documentation for classifications, data masking policies, and raw access policies was available, though RBAC design was simpler due to the monorepo setup. They discussed the need for a solution design document and migration guide for downstream consumers. Mazen inquired about the number of current DBT models, which Kaihao estimated at around 50, and asked about the DBT version and platform being used.
DBT Core Data Architecture Review
The team discussed the use of DBT Core for data transformation and the current state of the data architecture, with Kaihao noting that they primarily focused on payments and do not have a comprehensive architecture diagram. Mazen inquired about dependencies and table consumption, to which Kaihao confirmed they mainly use foundational domain tables. They discussed the potential need for views to maintain continuity for downstream consumers during a full redesign, with the decision to provide guidance for migration rather than recreating views depending on the extent of changes. Mazen also confirmed that all source data is currently available in Snowflake, and they briefly touched on the monetization domain and a missing supporting document for a specific pipeline.
Subscription Data Migration Discussion
The team discussed additional work needed for subscription data migration, which is separate from the main monetization domain project. Kai explained that this involves data type changes and outstream model rebuilds, while Ellen clarified that Canva prefers to keep complex subscription work in-house due to the high contextual knowledge required. The team agreed to have a separate discussion with Aiden when he returns from leave next week to further flesh out the details of the subscription work. Mazen noted that the current documentation was unclear about these additional tasks.
Project Scope and Data Transformation
Mazen and Kaihao discussed the scope of their project, focusing on monetization and data transformation tasks. They clarified that only the first two items, merchant fee reporting and payout offer coupon, would be addressed in the current scope. Mazen agreed to email Aiden about the third item and copy the team for further discussion. They also talked about performance data quality and compliance validation requirements, which Kaihao explained would involve efficient model redesign, incremental builds, and unit and integration testing. The team discussed the level of assistance needed from Snowflake, including solution design, analysis, and hands-on implementation, with Kaihao emphasizing the importance of testing for the new models.
Data Validation Solution Development
Kaihao and Mazen discussed the development of an end-to-end solution for data validation and metrics definition, emphasizing the need for an integrated pipeline rather than separate discussions. They agreed on including productionization in the process, with a focus on adhering to the existing go-to-production process. The current data volume was estimated to be in the gigabytes, and while some metric definitions exist for the merchant fee reporting, others need to be developed. Mazen inquired about the current data models, confirming they exist in a monolithic solution.
Dbt Model Complexity Assessment
The team discussed the complexity of dbt models across different domains, with the monetization domain rated as 6-7 out of 10 compared to other domains. Mazen requested sample code to assess complexity, but the team agreed they could not share actual code due to compliance concerns. Vanessa suggested working with Frank and Stephen to establish a compliant way to share sample data through Canva's platform, with Vish having appropriate access rights.
DBT Model Complexity Categorization
The team discussed categorizing DBT models as simple, medium, or complex, with Mazen explaining that complex models use macros and CTEs repeatedly, while simple models are direct select statements. Kaihao agreed to analyze the model lineage and provide a breakdown of model complexity percentages, while Mazen noted that based on statistical norms, most models would likely fall in the medium category. The team confirmed that all target tables are Snowflake native tables with no transient or external tables, and discussed the refresh frequency, which appears to be a combination of event-driven and schedule-based triggers.
Kubernetes to Airflow Transition Plan
The team discussed transitioning from a Kubernetes job to Airflow for data processing, with Kaihao explaining that the initial implementation will be event-based and daily, while the rest will be batch mode. They confirmed that existing data quality tests in DBT and YAML files will be migrated, and there will be a 3-month parallel run with only one version of the foundation. Mazen raised a concern about ensuring proper triggering between the old and new pipelines during the parallel run, and Kaihao confirmed that the data platform team will provide event-based jobs for Airflow.
dbt Migration Strategy Discussion
The team discussed the migration of dbt models from a monolith to a new system, with a 3-month transition period allowing users to access both old and new models. Kaihao explained that while row counts might differ between the old and new systems due to model redesigns, data validation should remain consistent. The team confirmed that existing dbt YAML files serve as primary documentation for the models, with some business documentation available on Confluence. They agreed that while there won't be a complete freeze on model changes, any modifications during the migration period need to be communicated to ensure data consistency.
